# SQL 문법 정리

## 조건에 맞는 도서와 저자 리스트 출력하기

### 사용한 SQL

```sql
SELECT
    b.BOOK_ID,
    a.AUTHOR_NAME,
    DATE_FORMAT(b.PUBLISHED_DATE, '%Y-%m-%d') AS PUBLISHED_DATE
FROM BOOK AS b
JOIN AUTHOR AS a
    ON b.AUTHOR_ID = a.AUTHOR_ID
WHERE b.CATEGORY = '경제'
ORDER BY b.PUBLISHED_DATE;
```

### 문제 이해하기

이 문제는 `BOOK` 테이블과 `AUTHOR` 테이블을 함께 사용해야 한다.

`BOOK` 테이블에는 책 정보가 들어 있고, `AUTHOR` 테이블에는 저자 정보가 들어 있다.

문제에서 필요한 값은 다음과 같다.

| 출력 컬럼 | 가져오는 테이블 |
| --- | --- |
| `BOOK_ID` | `BOOK` |
| `AUTHOR_NAME` | `AUTHOR` |
| `PUBLISHED_DATE` | `BOOK` |

즉, 책 정보만 보면 `AUTHOR_NAME`을 알 수 없고, 저자 정보만 보면 책의 출판일을 알 수 없다.
그래서 두 테이블을 연결하는 `JOIN`이 필요하다.

### 핵심 문법

#### 1. FROM

```sql
FROM BOOK AS b
```

`FROM`은 조회를 시작할 기준 테이블을 정한다.

이 문제에서는 책 목록을 조회하는 문제이므로 `BOOK` 테이블을 기준으로 잡는다.

`AS b`는 `BOOK` 테이블에 `b`라는 별명을 붙인 것이다.

```sql
BOOK AS b
```

이렇게 별명을 붙이면 이후에 컬럼을 쓸 때 다음처럼 짧게 작성할 수 있다.

```sql
b.BOOK_ID
b.AUTHOR_ID
b.PUBLISHED_DATE
```

여기서 `b.BOOK_ID`는 `BOOK` 테이블의 `BOOK_ID`라는 뜻이다.

#### 2. JOIN

```sql
JOIN AUTHOR AS a
```

`JOIN`은 두 테이블을 연결해서 하나의 결과처럼 조회할 때 사용한다.

이 문제에서는 `BOOK` 테이블에 있는 책 정보와 `AUTHOR` 테이블에 있는 저자 이름을 같이 출력해야 한다.

그래서 `BOOK` 테이블에 `AUTHOR` 테이블을 붙인다.

```sql
FROM BOOK AS b
JOIN AUTHOR AS a
```

이 코드는 말로 풀면 다음과 같다.

```text
BOOK 테이블을 기준으로 조회하되,
AUTHOR 테이블도 함께 연결해서 사용하겠다.
```

#### 3. ON

```sql
ON b.AUTHOR_ID = a.AUTHOR_ID
```

`ON`은 두 테이블을 어떤 기준으로 연결할지 정하는 부분이다.

`BOOK` 테이블에도 `AUTHOR_ID`가 있고, `AUTHOR` 테이블에도 `AUTHOR_ID`가 있다.

| BOOK 테이블 | 의미 |
| --- | --- |
| `AUTHOR_ID` | 이 책을 쓴 저자의 ID |

| AUTHOR 테이블 | 의미 |
| --- | --- |
| `AUTHOR_ID` | 저자를 구분하는 ID |
| `AUTHOR_NAME` | 저자 이름 |

따라서 두 테이블은 `AUTHOR_ID`를 기준으로 연결할 수 있다.

예를 들어 데이터가 다음과 같다고 해보자.

`BOOK`

| BOOK_ID | AUTHOR_ID | CATEGORY | PUBLISHED_DATE |
| --- | --- | --- | --- |
| 1 | 10 | 경제 | 2020-01-01 |
| 2 | 20 | 인문 | 2021-03-15 |

`AUTHOR`

| AUTHOR_ID | AUTHOR_NAME |
| --- | --- |
| 10 | 홍길동 |
| 20 | 김철수 |

`ON b.AUTHOR_ID = a.AUTHOR_ID`를 사용하면 다음처럼 같은 `AUTHOR_ID`끼리 붙는다.

| BOOK_ID | BOOK.AUTHOR_ID | AUTHOR.AUTHOR_ID | AUTHOR_NAME |
| --- | --- | --- | --- |
| 1 | 10 | 10 | 홍길동 |
| 2 | 20 | 20 | 김철수 |

즉, 책에 적힌 저자 ID를 보고 실제 저자 이름을 찾아오는 구조이다.

#### 4. SELECT

```sql
SELECT
    b.BOOK_ID,
    a.AUTHOR_NAME,
    DATE_FORMAT(b.PUBLISHED_DATE, '%Y-%m-%d') AS PUBLISHED_DATE
```

`SELECT`는 최종 결과로 보여줄 컬럼을 고르는 부분이다.

```sql
b.BOOK_ID
```

`BOOK` 테이블에서 책 ID를 가져온다.

```sql
a.AUTHOR_NAME
```

`AUTHOR` 테이블에서 저자 이름을 가져온다.

```sql
DATE_FORMAT(b.PUBLISHED_DATE, '%Y-%m-%d') AS PUBLISHED_DATE
```

출판일을 `YYYY-MM-DD` 형태로 출력한다.

프로그래머스 SQL 문제에서는 날짜가 `2021-01-01 00:00:00`처럼 시간까지 포함되어 나올 수 있다.
문제에서 `2021-01-01` 같은 형식을 요구하면 `DATE_FORMAT()`으로 형식을 맞춰주는 것이 안전하다.

#### 5. WHERE

```sql
WHERE b.CATEGORY = '경제'
```

`WHERE`는 조건에 맞는 행만 남길 때 사용한다.

이 문제에서는 카테고리가 `경제`인 도서만 출력해야 한다.

`CATEGORY`는 `BOOK` 테이블에 있는 컬럼이므로 별명 `b`를 붙여서 다음처럼 작성한다.

```sql
b.CATEGORY = '경제'
```

#### 6. ORDER BY

```sql
ORDER BY b.PUBLISHED_DATE;
```

`ORDER BY`는 결과를 정렬할 때 사용한다.

문제에서 출판일 기준 오름차순 정렬을 요구하므로 `PUBLISHED_DATE`를 기준으로 정렬한다.

오름차순은 기본값이기 때문에 `ASC`를 생략할 수 있다.

아래 두 코드는 같은 의미이다.

```sql
ORDER BY b.PUBLISHED_DATE;
```

```sql
ORDER BY b.PUBLISHED_DATE ASC;
```

### JOIN을 처음 볼 때 중요한 생각

`JOIN`은 어렵게 느껴질 수 있지만, 핵심은 하나이다.

```text
두 테이블에서 같은 의미를 가진 컬럼을 찾아서 연결한다.
```

이 문제에서는 두 테이블 모두 `AUTHOR_ID`를 가지고 있다.

그래서 다음 조건으로 연결한다.

```sql
ON b.AUTHOR_ID = a.AUTHOR_ID
```

이 조건이 없으면 SQL은 어떤 책과 어떤 저자를 연결해야 하는지 알 수 없다.

### 실행 순서로 이해하기

SQL은 작성 순서와 실제 처리 흐름이 조금 다르다.

이 문제의 흐름은 다음처럼 이해하면 좋다.

```text
1. BOOK 테이블을 기준으로 잡는다.
2. AUTHOR 테이블을 AUTHOR_ID 기준으로 연결한다.
3. CATEGORY가 '경제'인 책만 남긴다.
4. 필요한 컬럼 BOOK_ID, AUTHOR_NAME, PUBLISHED_DATE를 고른다.
5. 출판일 기준으로 오름차순 정렬한다.
```

### 최종 정리

이 문제의 핵심은 `BOOK` 테이블만으로는 저자 이름을 알 수 없다는 점이다.

그래서 `BOOK.AUTHOR_ID`와 `AUTHOR.AUTHOR_ID`를 기준으로 두 테이블을 `JOIN`하고, 연결된 결과에서 저자 이름을 가져온다.

```sql
JOIN AUTHOR AS a
    ON b.AUTHOR_ID = a.AUTHOR_ID
```

이 부분이 문제의 가장 중요한 문법이다.

### 다른 풀이 방식

같은 문제를 꼭 한 가지 방식으로만 풀 수 있는 것은 아니다.

이 문제는 결국 `BOOK` 테이블에서 조건에 맞는 책을 찾고, `AUTHOR` 테이블에서 저자 이름을 가져오면 된다.
그래서 테이블을 연결하는 방식에 따라 몇 가지 풀이가 가능하다.

#### 1. INNER JOIN을 명시하는 방식

```sql
SELECT
    b.BOOK_ID,
    a.AUTHOR_NAME,
    DATE_FORMAT(b.PUBLISHED_DATE, '%Y-%m-%d') AS PUBLISHED_DATE
FROM BOOK AS b
INNER JOIN AUTHOR AS a
    ON b.AUTHOR_ID = a.AUTHOR_ID
WHERE b.CATEGORY = '경제'
ORDER BY b.PUBLISHED_DATE;
```

`JOIN`만 작성해도 기본적으로 `INNER JOIN`과 같은 의미이다.

즉, 아래 두 코드는 같은 방식으로 동작한다.

```sql
JOIN AUTHOR AS a
```

```sql
INNER JOIN AUTHOR AS a
```

`INNER JOIN`은 두 테이블에서 연결 조건이 맞는 행만 결과로 가져온다.

이 문제에서는 다음 조건이 맞는 책과 저자만 연결된다.

```sql
b.AUTHOR_ID = a.AUTHOR_ID
```

처음 `JOIN`을 공부할 때는 `INNER JOIN`이라고 명확히 적는 것도 좋다.
두 테이블에서 조건이 일치하는 데이터만 가져온다는 의미가 더 잘 보이기 때문이다.

#### 2. WHERE에서 테이블을 연결하는 방식

```sql
SELECT
    b.BOOK_ID,
    a.AUTHOR_NAME,
    DATE_FORMAT(b.PUBLISHED_DATE, '%Y-%m-%d') AS PUBLISHED_DATE
FROM BOOK AS b, AUTHOR AS a
WHERE b.AUTHOR_ID = a.AUTHOR_ID
  AND b.CATEGORY = '경제'
ORDER BY b.PUBLISHED_DATE;
```

이 방식은 `JOIN ... ON`을 쓰지 않고 `WHERE`에서 두 테이블을 연결한다.

```sql
FROM BOOK AS b, AUTHOR AS a
```

위 코드는 `BOOK` 테이블과 `AUTHOR` 테이블을 함께 사용하겠다는 뜻이다.

그리고 아래 조건으로 두 테이블을 연결한다.

```sql
WHERE b.AUTHOR_ID = a.AUTHOR_ID
```

여기에 카테고리 조건도 함께 작성한다.

```sql
AND b.CATEGORY = '경제'
```

결과는 `JOIN ... ON`을 사용한 풀이와 같다.

하지만 이 방식은 추천도가 조금 낮다.
테이블을 연결하는 조건과 데이터를 필터링하는 조건이 모두 `WHERE`에 섞이기 때문이다.

테이블이 2개일 때는 괜찮지만, 테이블이 3개 이상으로 늘어나면 어떤 조건이 연결 조건이고 어떤 조건이 필터 조건인지 헷갈리기 쉽다.

그래서 보통은 아래처럼 `JOIN ... ON`으로 연결 조건을 분리해서 작성하는 방식이 더 읽기 좋다.

```sql
FROM BOOK AS b
JOIN AUTHOR AS a
    ON b.AUTHOR_ID = a.AUTHOR_ID
WHERE b.CATEGORY = '경제'
```

#### 3. 서브쿼리로 저자 이름을 가져오는 방식

```sql
SELECT
    b.BOOK_ID,
    (
        SELECT a.AUTHOR_NAME
        FROM AUTHOR AS a
        WHERE a.AUTHOR_ID = b.AUTHOR_ID
    ) AS AUTHOR_NAME,
    DATE_FORMAT(b.PUBLISHED_DATE, '%Y-%m-%d') AS PUBLISHED_DATE
FROM BOOK AS b
WHERE b.CATEGORY = '경제'
ORDER BY b.PUBLISHED_DATE;
```

이 방식은 `JOIN`을 직접 쓰지 않고, `SELECT` 안에서 저자 이름을 따로 조회한다.

핵심 부분은 다음이다.

```sql
(
    SELECT a.AUTHOR_NAME
    FROM AUTHOR AS a
    WHERE a.AUTHOR_ID = b.AUTHOR_ID
) AS AUTHOR_NAME
```

이 코드는 말로 풀면 다음과 같다.

```text
현재 조회 중인 책의 AUTHOR_ID와 같은 AUTHOR_ID를 가진 저자를 찾아서,
그 저자의 AUTHOR_NAME을 가져온다.
```

예를 들어 `BOOK` 테이블에서 현재 보고 있는 책의 `AUTHOR_ID`가 `10`이라면, 서브쿼리는 `AUTHOR` 테이블에서 `AUTHOR_ID = 10`인 저자를 찾는다.

그리고 그 저자의 이름을 `AUTHOR_NAME` 컬럼으로 출력한다.

이 방식은 책 한 권마다 저자 이름을 따로 찾아오는 느낌이라 처음에는 이해하기 쉬울 수 있다.

하지만 이런 유형의 문제에서는 보통 `JOIN`을 사용하는 방식이 더 자연스럽다.
두 테이블의 관계를 명확하게 보여주고, 여러 컬럼을 함께 가져올 때도 확장하기 쉽기 때문이다.

### 풀이 방식 비교

| 방식 | 특징 | 추천도 |
| --- | --- | --- |
| `JOIN ... ON` | 가장 일반적이고 읽기 좋다 | 높음 |
| `INNER JOIN ... ON` | `JOIN`의 의미를 더 명확히 쓴다 | 높음 |
| `FROM A, B WHERE ...` | 예전 스타일의 조인 방식이다 | 낮음 |
| 서브쿼리 | JOIN 없이 값을 가져올 수 있다 | 보통 |

이 문제에서는 아래 방식이 가장 추천된다.

```sql
FROM BOOK AS b
JOIN AUTHOR AS a
    ON b.AUTHOR_ID = a.AUTHOR_ID
```

처음에는 `JOIN`이 낯설 수 있지만, 결국 핵심은 다음 한 문장이다.

```text
같은 의미를 가진 컬럼끼리 연결해서 두 테이블의 정보를 함께 조회한다.
```
